AI Agent 入   门指南    ——	第 1 页



第一章  	什 么是 AI Agent



AI Agent（智能体）  	是一   种
能够   自主  感
知环境、做出决策并  执
行行动的   软件系统。与传   统的聊天机器人  不同，Agent 不只是"一   问一答"，而是能   够主动规划多步  操
作，调用   外部工具，在失败时   自我纠错。


Agent 的   四大核   心能力：



1. 感   知（Perception）：	从用户   输入、API	返回结果、文件内   容中提取关
键信息。感知是 Agent 的"眼睛和耳朵"，没有感   知能力，Agent 就无
法理   解外部世界   发生了什么。

2. 推   理（Reasoning）：基于大语   言模型分析问题，将复   杂任务拆解为可执行的子
任务。推理能力决定了 Agent 解决问   题的深度和   广度。

3. 执   行（Action）：调用搜索   工具、执行代
码、查询数据库   来完成实际工作。执行让 Agent 从"会说"变成"能   做"。

4. 记   忆（Memory）：维护短期对   话上下文和长期知识库。记忆让 Agent 能够进   行连贯的多轮对话，记住用户的偏好   和历史决策。



第二章  RAG 检索增   强生成



RAG（Retrieval-Augmented	Generation）是 Agent 获取外部   知识的核
心手段。它解决了 LLM 的两   个致命缺陷：知识有截   止日期，且无法访问   私域数据。



RAG 的工作   流程分为两   个阶段：



离线索引（提   前完成，一次索引多次查询）：	文档解析   →   文本分块   →   向量化   →   存入   向量库

在线查询（每   次提问都执行）：	问题向量化   →   检索相似   片段   →   拼入 Prompt   →   LLM 生成   回答



RAG 的关键优   势：

•   不占用上下   文窗口——只取最相关的   片段，而非塞入   整篇文档

•   知识可随时更   新——重新索引文   档即可，无需重新训   练模型

•   答案可溯   源——能明确指出信   息来自哪个文档的   哪个片段

•   降低幻觉——让 LLM "查资料回答"而非"凭记忆   回答"



第三章  Function Calling 工   具调用



Function	Calling 让 Agent 从	"只会说话"进化到"能   动手"。这是 Agent 区别于聊天机器   人的核心特征。



标准的四   步流程：



第一步：客   户端在请求中声明   可用工具，包括函数名、参   数 schema、功能   描述

第二步：模   型根据用户意图决定   是否调用工具，返回	tool_calls

第三步：客户端   执行对应的函数，获取执   行结果

第四步：将执行   结果封装为 role=tool 的消息，回传   给模型继续处理



重要安全   设计：模型本身不执行   任何代码，执行权完   全掌握在客户端手中。这防止了模型直接   操作文件系统、网络等敏感   资源。



第四章  Harness	Loop 执行   控制循环



Harness Loop 是 Agent 的执行内   核，定义了 Agent 如何持续   运行、做出决策并完成   任务。没有 Harness Loop，Agent 就是单次   问答，不具备自主执   行能力。



标准 Harness	Loop 五个步   骤：



1. 读取当前   状态（state）：包括对   话历史、工具返回结果、中   间产出

2. 交给 LLM 做决   策：LLM 根据当前状   态决定下一步做什么

3. 解析 LLM 的输   出：判断是文本回复   还是工具调用

4. 执行解析出的动   作：如果是工具调用就执   行，如果是文本就返回   给用户

5. 将执行结果   写回状态：更新 state，进入下一轮   循环



三种终止   条件：done 表示任务   已完成，max_steps 防止无   限循环（默认 15 步），error 表示执行出错需要   人工介入。



第五章  ReAct	推理模式



ReAct（Reasoning	+ Acting）是 Harness Loop 的一种增强模   式，核心思想是：行动   之前先思考。



ReAct 的每一   步都要求模型先输出 Thought（思考过程），再输出 Action（具   体行动）。这让 Agent 的决策过程变得   可见、可调试，显著提高   了复杂任务的完成率和准   确率。



传统的编程流程是写   死的：if-else、循环、函数调用，所   有路径在编码时就已   确定。Agent 的流程是 LLM 每步实时推   理出来的——它能处理未曾预见的   场景，但也带来了可控性和可调试性的   挑战。



第六章  Embedding 与向   量检索



Embedding（嵌入）是把   文本映射为高维向量的技   术。一段文字经过 Embedding 模
型，变成一个固定长度的   浮点数数组（向量）。



这个向量就是文本在"语   义空间"中的坐标——语义相   近的文本，坐标也相近。"找最   相关的内容"因此变成了"找最近   的向量"。



本项目使用   的 GLM embedding-3 模型输出	2048	维向量。余弦   相似度用于衡量两个向量的方向一   致性，值域为 -1 到 1，越接近 1	表示语义越相近。



向量检索的基本流   程：将 query 向量化，与向量   库中所有向量逐一计算余弦   相似度，按分数降序排列，取 topK	条结果。



第七章  文   本分块策略



文本分块（Chunking）是 RAG 的关键预   处理步骤。分块粒度直接   决定"能检索到什么"和"拼给 LLM 的   上下文是否有意义"。



三种分块策略   的对比：

•   太碎（100 字/块）：检索精准但上下   文不足，LLM 无法生成完整   答案

•   太粗（1500 字/块）：上下文完整但主   题混杂，检索精度   下降

•   甜点区（300-500 字/块）：话题聚焦、上下文充足，三项检   索指标（准确率、完整度、区分度）均表   现最优



固定长度分块 + 重叠窗   口（overlap）是基础策略，能防止语义在边界被硬切   断。更高级的策略包括按语义边   界分块和递归分块。



第八章  PDF 解析与   文本清洗



PDF 解析是 RAG 的"入口"——Garbage	In, Garbage Out。



PDF 是排版格式而非文本格   式，提取的文本常带多余换   行、断行连字符、页眉页脚   碎片、连续空格。这些"脏数据"如果不   清洗就直接分块，会导致：

•   完整句子被垃圾字符切碎，检   索不到

•   向量库中大量"噪音块"，降低整体   检索精度

•   LLM 收到残缺片段，无法   生成准确答案



文本清洗策略：统一   换行符 → 折叠连续空行（保留段落结构）→ 压缩连续空格 → 去除首尾   空白。

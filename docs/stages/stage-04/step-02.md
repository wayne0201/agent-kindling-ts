# Step 2: 文本分块（Chunking）—— 知识的切分艺术

## 💡 先说人话

**RAG 的第一步是把知识库"撕成小纸条"。**

纸条撕多大直接决定了后面能不能找到正确答案：

- 撕太碎（每张一句话）→ 纸条太小，光看一张不知道在说什么
- 撕太大（每张一整章）→ 什么话题都混在一起，想找"函数调用"翻到的却是"什么是 LLM"
- **400 字一张、每张跟前后各重叠一小段**——这是最合适的撕法

撕好纸条，后面的检索才能"翻对页"。撕得不好，再先进的检索也找不准。

## 🎯 本环节目标

实现"固定长度 + 重叠窗口"分块策略，通过**同一个文档 × 同一条 query × 三种分块策略**的完整对比实验，建立"分块粒度 = 检索精度"的直觉。本步是纯本地逻辑不调 API（第一部分），第二部分会实际调用 Embedding API 做检索对比。

实验分三个部分：
1. **切分形态**：看不同参数切出来的 chunk 长什么样
2. **检索对比**：同样 query 在三种策略下检索 top-3，评判准确率/完整度/区分度
3. **综合汇总**：哪项策略三项全过

## 💡 核心原理速记

- **为什么需要分块**：① Embedding 模型有输入上限；② 整篇一个向量太粗（找不到细节），按句太细（没有上下文）；③ 生成时拼 Prompt 受 token 预算约束。详见 [文本分块](../../concepts/rag/chunking.md)。
- **怎么分**：按固定字符数切，每相邻两张纸条之间留一小段重叠（overlap），防止一句话被切在两块边界上。
- **怎么判断分得好不好**：用准确率、完整度、区分度三个指标来评判（见下文）。
- **甜点区**：300-500 字符。太粗（1500）主题混杂，太细（100）信息碎片化。

## 💻 落地内容与代码

**代码实现**: [demos/stage-04/step-02/chunking.ts](../../demos/stage-04/step-02/chunking.ts)

**核心逻辑**:

```
splitText(text, chunkSize, overlap)
  │
  ├─ while 循环，游标从 0 开始
  │    ├─ 切 chunkSize 个字符 → push 一个 chunk
  │    ├─ 游标回退 overlap 个字符（让下一块跟前一块有重叠）
  │    └─ 重复直到文本末尾
  │
  └─ 返回 Chunk[]（每条含 id / text / start / end）
```

**检索 + 评判 pipeline**（`evaluate` 函数）：

```
query → embed() → query向量
                         │
chunks → embed() → 每个chunk的向量 ─┤ 逐一算余弦相似度
                                    │
                                    ▼
                              排序取 top-3
                                    │
                        ┌───────────┼───────────┐
                        ▼           ▼           ▼
                     准确率       完整度       区分度
                  (正确答案在  (top-1 ≥200字?  (top1 - top3
                   top-3里吗?)    够LLM答题?)    > 0.05?)
```

**三项评判指标**：

| 指标 | 含义 | 计算方法 |
|------|------|----------|
| **准确率** | 正确答案的 chunk 在不在 top-3？ | top-3 文本中是否包含预期关键词 |
| **完整度** | top-1 的上下文够不够 LLM 答题？ | top-1 chunk 原文 ≥ 200 字符 |
| **区分度** | 相关和不相关的分数拉开了吗？ | top1.score - top3.score > 0.05 |

**代码亮点**:
- 三组策略对比 + 三个 query 交叉验证 = 9 组数据点，综合评判一目了然
- "短 chunk 高分陷阱"通过准确率/完整度的矛盾揭示：太碎得分最高（0.57）但完整度零分，实际不可用

## 🐞 踩坑记录与思考

- **注意**: `overlap` 必须小于 `chunkSize`，否则死循环。
- **注意**: 短 chunk 余弦相似度天然偏高——不要被高分骗了，看 top-1 内容是否完整。
- **注意**: 固定长度切分不考虑语义边界，可能切断句子。Stage 5 会引入按语义边界切分。
- **思考**: 实际项目调参方法——准备 5 个已知答案的 query，跑 `evaluate()` 看三项指标，每次只调一个参数 20-30%，重新索引再测。

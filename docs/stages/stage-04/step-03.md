# Step 3: PDF 解析与文本清洗

## 💡 先说人话

**PDF 提取出来的文字是"脏"的——就像从洗衣机里掏出来的衣服，不拧干没法穿。**

真实 PDF 提取的文字长这样：句子中间乱换行、页眉页脚碎片混进来、连续十几个空格、一个词被切成两半。这些"脏东西"如果不洗掉，后面撕纸条（分块）时会把完整句子切得七零八落，检索时自然找不到。

本步用一份精心构造的"脏文档"（约 3000+ 字符，含大量垃圾字符），清洗后丢进向量库做检索对比。实测数据：净文档的检索命中分数 +5.5%，区分度 +42%。**Garbage In, Garbage Out——这是可量化的，不是口号。**（具体数值因脏文档内容而异，以实际运行输出为准）

## 🎯 本环节目标

用 `pdf-parse` 从 PDF 提取纯文本，实现文本清洗管线。通过一份"脏文档"的清洗前后检索对比实验，量化验证"洗菜"对 RAG 检索质量的影响。

实验分三部分：
1. **清洗前后可视化**：脏文档长什么样 vs 净文档长什么样
2. **检索质量对比**：5 条 query × 脏净文档 → 准确率 / 完整度 / 区分度 / 命中分数四项指标
3. **综合评判**：四项指标汇总，量化垃圾对检索的损害

## 💡 核心原理速记

- **PDF 提取 = 脏数据**：PDF 是排版格式不是文本格式，提取的文字带多余换行、断行连字符、页眉页脚、连续空格和 tab。
- **清洗五步**：统一换行（`\r\n` → `\n`）→ 折叠空行（3+ → 2 个）→ 压缩空格（`[ \t]+` → 单空格）→ 去行尾空格（` +\n` → `\n`）→ 去首尾空白（`.trim()`）。
- **脏数据如何损害检索**：垃圾字符撑大文档体积（约 10-15% 垃圾）→ 多出噪音 chunk → 相似度被稀释 → 区分度大幅下降 → 正确答案排不进 topK。
- **动态导入保底**：`pdf-parse` 动态 import，未安装时优雅降级而非崩溃。

## 💻 落地内容与代码

**代码实现**: [demos/stage-04/step-03/parse-pdf.ts](../../demos/stage-04/step-03/parse-pdf.ts) · 共享模块 [_shared/pdf-parser.ts](../../demos/stage-04/_shared/pdf-parser.ts)

**学习资源**: [dirty-sample.txt](../../demos/stage-04/knowledge/dirty-sample.txt) — 精心构造的脏文档，含 183 处连续空格 + 36 处连续空行 + 大量 tab。

**实验 pipeline**：

```
dirty-sample.txt (3224 字符, 14.3% 垃圾)
        │
        ├──→ 脏文档 → splitText → 11 chunks → embed → 向量库 A
        │                                                   │
        └──→ cleanText → 净文档 (2762 字符)                 │
                │                                           │
                └──→ splitText → 10 chunks → embed → 向量库 B
                                                            │
                    5 条 query ──→ 同时检索 A 和 B ─────────┘
                                    │
                    准确率 / 完整度 / 区分度 / 命中分数
```

**四项评判指标**：

| 指标 | 含义 | 清洗效果（实测数据）|
|---|---|---|
| **准确率** | 正确答案在 top-3 里吗？ | 脏净同等 |
| **完整度** | top-1 chunk 上下文够 LLM 答题吗？ | 脏净同等 |
| **区分度** | 相关/不相关 chunk 分数拉开了吗？ | 净文档显著更高 |
| **命中分数** | 检索对正确答案有多"确信"？ | 净文档 +5% 以上 |

**代码亮点**:
- 脏文档由 8 章内容 + 大量空格/tab/换行构成，模拟真实 PDF 提取效果
- 5 条 query 覆盖全部 8 章，避免单条 query 的偶然性
- 四项指标从不同维度量化清洗收益，不是"洗了就干净"而是"洗了检索真的变好"

## 🐞 踩坑记录与思考

- **注意**: `pdf-parse` 无官方 TypeScript 类型，需自建 `.d.ts`（`_shared/pdf-parse.d.ts`）。
- **注意**: 关键词匹配用 80 字 preview 会截断关键词，已改为 fullText 匹配。
- **注意**: 小文档（< 5000 字）检索差异不明显——因为 embedding 的 2048 维足够"宽容"。真实 PDF（> 50000 字）垃圾累积效应才会让准确率崩盘。
- **思考**: 区分度是比准确率更敏感的指标——文档不够脏时准确率看不出差距，但区分度已经暴露了信号稀释。实际项目评估清洗效果，优先看区分度和命中分数，而非只看"找到没找到"。

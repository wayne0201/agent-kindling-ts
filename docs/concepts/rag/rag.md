# 概念：RAG（检索增强生成）

## 💡 先说人话

**RAG 就是"不塞整本书，只递最相关的几页纸条"。**

想象你是一个答题的人（LLM），面前有一本厚厚的手册。有人来问问题，你不可能把整本书塞进脑子里（上下文窗口不够大），也不可能全凭记忆乱猜（会产生幻觉）。

RAG 的做法：**提前把书撕成小纸条，每张纸条写一个话题，按话题归类放好。有人提问时，翻出最相关的几张纸条递给你，你照着纸条上的内容回答。**

```
📦 入库（提前做一次）                    🔍 查询（每次提问时）
─────────────────────                    ─────────────────────
文档 → 撕成纸条 → 给每张纸条                 问题 → 也变成"坐标"
      贴上"坐标" → 存进库                   → 跟库里所有纸条比谁更像
                                            → 取最像的3张
                                            → 拼进 Prompt 给 LLM
                                            → LLM 照着纸条回答
```

**这样既不塞整本书占用窗口，也不凭记忆瞎猜产生幻觉。** 用多少取多少——这就是 RAG 和"把文档全塞进上下文"的核心区别：前者是按需检索，后者是暴力填充。

## 📖 一句话定义

RAG = Retrieval-Augmented Generation = 检索 + 生成，让 LLM 从"凭记忆回答"升级为"查资料回答"——在生成前先从外部知识库检索相关片段拼进 Prompt，解决模型知识截止与私域数据访问问题。

## ⚙️ 深入原理

### RAG 完整流程

```
┌──────────── 离线索引（一次） ────────────┐    ┌──────── 在线查询（每次） ────────┐
│                                          │    │                                  │
│  文档 → 分块 → 向量化 → 入向量库         │    │  query → 向量化                  │
│         (chunking)  (embedding)  (store) │    │     ↓                            │
│                                          │    │  与库中所有向量算余弦相似度      │
│                                          │    │     ↓                            │
│                                          │    │  取 topK 片段                    │
│                                          │    │     ↓                            │
│                                          │    │  拼进 System Prompt 作为【资料】 │
│                                          │    │     ↓                            │
│                                          │    │  LLM 基于资料生成回答            │
└──────────────────────────────────────────┘    └──────────────────────────────────┘
```

左侧是离线索引（文档入库，做一次即可持久化复用），右侧是在线查询（每次提问都跑一遍）。下面是 RAG 查询的概念实现（含来源和相似度，实际 Step 5 代码更精简）：

```typescript
async function ragQuery(store: VectorStore, question: string): Promise<RagAnswer> {
  // 1. 检索：query 向量化 → 算相似度 → 取 topK
  const sources = await store.search(question, TOP_K);

  // 2. 拼接上下文：把 topK 片段拼成【检索资料】
  const context = sources
    .map((r, i) => `[片段 ${i + 1}]（来源: ${r.record.metadata.source}，相似度: ${r.score.toFixed(3)}）\n${r.record.text}`)
    .join('\n\n');

  // 3. 构造 Prompt：把资料塞进 System Prompt
  // 4. 生成：LLM 基于资料回答
  const messages = [
    { role: 'system', content: `你是一个严谨的问答助手。请根据下方【检索资料】回答用户问题。
规则：
1. 只能基于【检索资料】回答，不要编造资料中没有的内容。
2. 如果【检索资料】中没有答案，明确回答"根据已有资料无法回答该问题"。
3. 回答时引用资料来源（如"根据片段 1..."）。

【检索资料】
${context}` },
    { role: 'user', content: question },
  ];
  const response = await client.chat.completions.create({ model: MODEL, messages });
  return { answer: response.choices[0].message.content ?? '', sources };
}
```

### 与微调对比

| 维度 | RAG | 微调（Fine-tuning） |
|------|-----|---------------------|
| 解决的问题 | 知识获取（知道什么） | 行为模式（怎么说/怎么答） |
| 数据更新 | 重新索引即可，分钟级 | 重新训练，小时~天级 |
| 成本 | 检索 + Prompt 增量 | 训练 + 托管定制模型 |
| 可溯源 | ✅ 能指出答案来自哪个片段 | ❌ 知识熔进权重，无法溯源 |
| 私域数据 | 存外部库，不进模型权重 | 进权重，有泄露风险 |

**关键认知**：RAG 和微调不是二选一。RAG 解决"模型不知道"，微调解决"模型不会"。需要模型掌握新事实、新文档——用 RAG；需要模型改变语气、格式、推理风格——用微调。本项目的 RAG 落地不碰模型权重，纯粹靠"检索 + Prompt 拼接"给模型"开卷考试"。

### 防幻觉的关键：System Prompt 约束

RAG 防幻觉的核心不在检索，而在 System Prompt 的强约束："**只根据资料回答，资料没有就说不知道**"。这条规则把 LLM 从"自由发挥"模式切换到"阅读理解"模式——它的任务从"回答问题"变成"从给定资料中找答案"。

没有这条约束，即便检索到了正确片段，LLM 仍可能"顺手"补几句资料里没有的内容（幻觉）。有了这条约束，LLM 在资料不全时会主动承认"无法回答"，而不是编造。

### 域内 vs 域外问题

RAG 的价值在域外问题上体现得最清楚：

- **域内问题**（知识库有答案）：检索到高相似度片段，LLM 基于片段准确回答。
- **域外问题**（知识库无答案）：检索到的片段相似度都很低，LLM 按 System Prompt 约束回答"根据已有资料无法回答该问题"——**拒绝编造**就是 RAG 相比裸 LLM 的核心增益。

本项目 Step 5 用"2024 年世界杯冠军是谁"这类域外问题验证这一点：裸 LLM 可能凭训练记忆瞎猜，RAG 则因检索不到相关资料而拒绝回答。

### 局限性：检索质量决定回答质量

RAG 不是银弹，它的天花板是检索质量：

- **检索不到** → 没有资料可拼 → LLM 只能说"不知道"（哪怕知识库里其实有答案，只是分块/向量化没让它被召回）。
- **检索错了** → 拼了无关资料 → LLM 基于错误资料给出看似合理实则跑题的回答（比"不知道"更危险）。
- **检索到但片段残缺** → LLM 看到半句话 → 答案不完整。

这些问题的优化（查询重写、重排序、混合检索）是 Stage 5+ 的主题。Stage 4 先把闭环跑通，理解"检索 + 生成"的协作机制。

## 🛠️ 在 Agent 架构中的作用

- **Agent 的外部长期记忆**：Stage 3 的 messages 数组是"工作记忆"，受上下文窗口约束、会被滑动窗口截断；RAG 提供的向量库是"外部长期记忆"，容量不受窗口限制，按需检索。两者构成 Agent 的分层记忆系统。
- **突破上下文窗口硬上限**：哪怕 128K 窗口也装不下一个企业知识库。RAG 让 Agent 不必把所有资料塞进窗口，而是"用到哪段调哪段"，把窗口留给真正相关的片段。
- **Stage 3 截断丢失记忆的解决方案**：Stage 3 的滑动窗口截断让 Agent "忘了"早期对话——这是工作记忆的固有缺陷。RAG 把需要长期记住的内容（文档、历史决策、用户偏好）索引到向量库，Agent 通过检索"想起来"，从"被迫遗忘"走向"主动召回"。

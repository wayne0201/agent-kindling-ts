# 概念：文本分块（Chunking）

## 💡 先说人话

**分块就是把一本书撕成小纸条，每张纸条上写一个话题。**

撕多大直接决定后面"翻纸条"能不能翻对：

- 🗑️ 撕太碎（每张 1 句话）→ 纸条太小，光看一张不知道在说什么，LLM 没法答题
- 🗑️ 撕太大（每张一整章）→ 什么话题都混在一起，问"函数调用"翻到的却是"什么是 LLM"
- ✅ **400 字一张，每张跟前后各重叠一小段**——话题聚焦、上下文完整、不会把一句话撕成两半

撕纸条是 RAG 的第一道工序，纸条撕得好，后面检索才能翻对页。撕得不好，再先进的 Embedding 也找不准。

## 📖 一句话定义

文本分块把长文档切分为合适粒度的片段（chunk），是 RAG 检索精度与生成质量的关键预处理步骤——分块粒度直接决定"能检索到什么"和"拼给 LLM 的上下文是否有意义"。

## ⚙️ 深入原理

### 为什么必须分块

三个硬约束逼着我们必须分块，而不是把整篇文档直接丢给系统：

1. **Embedding 模型有输入长度上限**：GLM embedding-3 约 8K tokens，超长文档根本无法一次性向量化。
2. **检索粒度**：整篇文档一个向量太粗——查询"Function Calling 的四个步骤"时，整篇手册的向量会被"稀释"，相似度不够突出；按句又太细，单句缺乏上下文，检索到了也拼不出完整答案。
3. **生成阶段的 token 预算**：检索到的片段要拼进 Prompt 喂给 LLM，片段过长会挤占上下文窗口、推高成本。

分块的本质是**在"检索精度"与"上下文完整性"之间找平衡**——一个 chunk 既要足够聚焦于一个语义点（检索能命中），又要足够完整（拼进 Prompt 后 LLM 能看懂）。

### 固定长度 + 重叠窗口策略

本项目实现最基础的策略：按固定字符数切分，相邻 chunk 之间保留一段重叠（overlap）。

```
原文：  |===============================|
chunk1: |======|                       |
chunk2:       |--(overlap)--|======|   |   ← 回退 overlap 字符再切
chunk3:                     |--(overlap)--|======|
```

核心循环：

```typescript
while (start < cleaned.length) {
  const end = Math.min(start + chunkSize, cleaned.length);
  chunks.push({ id: id++, text: cleaned.slice(start, end), start, end });
  if (end >= cleaned.length) break;
  start = end - overlap;   // 回退 overlap 个字符，形成重叠窗口
}
```

**overlap 的作用**：防止一个完整语义被硬切断在两个 chunk 的边界。比如"Function Calling 的第三步是客户端执行工具"这句话正好横跨 chunk1 末尾和 chunk2 开头，没有 overlap 的话，chunk1 只剩半句、chunk2 也只剩半句，两段都检索不到完整语义。有了 overlap，这句话会完整出现在 chunk2 中。

### chunkSize / overlap 参数影响

| 参数 | 过小 | 过大 |
|------|------|------|
| `chunkSize` | 缺上下文，检索到了也答不全 | 语义被稀释，检索精度下降，拼 Prompt 占预算 |
| `overlap` | 边界语义被切断 | 重复内容多，存储与检索成本上升，相似片段冗余 |

**甜点区**：`chunkSize = 300~500` 字符是经验上的常用区间——段落级粒度，既聚焦又完整。本项目 Step 5 的 RAG 闭环采用 `CHUNK_SIZE = 400, OVERLAP = 60`。

参数约束上必须校验：`chunkSize > 0`，`overlap ∈ [0, chunkSize)`——overlap ≥ chunkSize 会导致死循环（每次 start 都回退到上一次起点之前）。

### 边界切断问题

固定长度切分是"按字符数硬切"，完全不考虑语义边界——可能把一个句子从中间切断，把"上下文窗口"切成"上下文窗"和"口"。这是该策略的固有缺陷。

更高级的策略（按句号/段落等语义边界切分、递归分块）留给 Stage 5。本阶段先用最朴素的策略把管线跑通，理解"分块粒度 = 检索精度"这个核心关系。

### Chunk 的结构化记录

每个 chunk 不只存文本，还记录起止位置，便于溯源与调试：

```typescript
export interface Chunk {
  id: number;
  text: string;
  start: number;   // 在原文中的起始字符偏移
  end: number;     // 结束偏移
}
```

入库时还会附加 `metadata: { source, chunkIndex }`，让检索结果能回答"这段话来自哪个文件的第几个分块"——这是 RAG 可溯源性的基础。

### 调参方法论：怎么知道参数合不合理？

分块参数的调优不是玄学，有一套可操作的方法。不同项目的文档结构和查询模式确实不同，但评估框架是通用的。

**第一步：从默认值起步（覆盖 80% 场景）**

```
新项目默认: chunkSize = 400, overlap = 60
```

这个组合是段落级粒度，对大多数中文技术文档、产品手册、FAQ 都适用。先别急着调，跑一遍检索看效果。

**第二步：用"检索命中测试"诊断**

准备 5-10 个你**知道答案在文档哪里**的典型问题，逐个检索，看 topK 结果：

| 现象 | 诊断 | 调整方向 |
|---|---|---|
| 相关片段排不进 top3，被无关片段挤掉 | chunk 太大，语义被稀释 | **减小 chunkSize**（如 400 → 300） |
| topK 片段文本太短，LLM 看完说"信息不足" | chunk 太小，上下文不够 | **增大 chunkSize**（如 300 → 500）|
| 一个完整段落被切成两块，两块相似度都不高 | 硬边界切断了语义 | **增大 overlap**（如 60 → 100）|
| 检索结果有大量重复内容 | overlap 太大，冗余过多 | **减小 overlap**（如 100 → 50）|

**第三步：根据文档类型微调**

不同内容的"自然段落"长度不同，需要匹配：

| 文档类型 | 推荐 chunkSize | 原因 |
|---|---|---|
| 技术手册、API 文档 | 300-500 | 章节短、概念密集，段落级足够 |
| 法律合同、论文 | 500-800 | 长段落多，单句缺乏完整语义 |
| FAQ、对话记录 | 200-400 | 每条独立，但需要包含 Q+A 对 |
| 新闻、博客 | 400-600 | 段落适中，主题明确 |
| 代码库（按函数/类分块）| 不适用 | 代码不适合固定长度，需 AST 分块（Stage 5+） |

**第四步：迭代闭环**

```
实现分块 → 跑检索命中测试 → 看 topK 日志 → 微调参数 → 重新索引 → 再测试
                                    ↑                                    │
                                    └────────────────────────────────────┘
```

关键原则：**不要一次调太多参数**。每次只改 chunkSize 或 overlap 中的一个，改 20-30%，重新索引后对比。如果两个同时改，你不知道哪个改变起了作用。

**常见误区**

> ❌ "越小越精准"——chunkSize=50，每条检索都精准命中，但 LLM 看完说"信息不足无法回答"
>
> ❌ "越大越完整"——chunkSize=2000，把整章塞进一个向量，检索时永远只能命中那一个 chunk
>
> ✅ **粒度要匹配你的 query 的"答案单元"大小**。如果你预期的问题答案是 2-3 句话（约 100-200 字），chunk 应该包含 3-5 句话（约 300-500 字），给 LLM 留足上下文余量。

### 警惕"高分陷阱"：短文本天然相似度偏高

一个反直觉的现象：chunkSize=100 的碎片，余弦相似度分数往往**高于** chunkSize=400 的完整段落。实测数据：

```
Query: "Function Calling 的四个步骤是什么？"

chunkSize=100  Top-1: 0.5721  → "temperature 参数控制输出的随机性…"  ❌ 毫不相关
chunkSize=400  Top-1: 0.5209  → "Function Calling 让 LLM 进化到能动手…流程分四步"  ✅ 正确答案
```

**为什么短文本分数更高？** 短 chunk 包含的语义维度少，query 向量往任意方向投影都不会"撞到"太多无关语义，分数自然偏高。但这恰恰是问题——高分不等于有用：

-  100 字碎片得了 0.57，但内容是错的，LLM 无法回答
-  400 字节落得了 0.52，但内容完整正确，LLM 能给出好答案

**评估分块质量，不能只看相似度数值，要看"高分 chunk 里有没有可用答案"。** 找的是"正确答案排进 topK"，不是"所有分数都高"。如果看到太碎的策略分数异常高，去看高分片段的具体内容——大概率是噪音。

## 🛠️ 在 Agent 架构中的作用

- **RAG 管线的入口**：文档 → **分块** → 向量化 → 入库 → 检索 → 生成。分块是整条管线的第一道工序，决定了后续所有环节处理的基本单元。
- **Garbage In, Garbage Out**：分块质量直接决定检索质量。切得太碎，检索到的片段拼不出完整答案；切得太粗，相似度被稀释，相关片段排不进 topK。分块策略的微小调整，会让最终回答质量出现肉眼可见的差异。
- **分块质量决定检索质量**：Agent 的"外部记忆"是否好用，一半取决于分块。Stage 3 的滑动窗口截断是"工作记忆"管理，RAG 是"外部记忆"管理，而分块是外部记忆的"编码格式"——编码格式不对，再先进的检索也召回不了正确内容。
